6. Billing architecture with Paystack

This part must be server-controlled.

Core billing rule

The frontend may start a payment, but the backend is the only trusted source that activates a plan.

Billing flow

User opens billing page in dashboard

User chooses plan

Frontend calls backend:
POST /billing/initialize-paystack

Backend creates Paystack transaction

Paystack checkout opens

User pays

Paystack redirects back to app

Paystack also sends webhook to backend

Backend verifies webhook and transaction

Backend updates subscription in Supabase

User plan becomes active

Important

Never trust the redirect alone.
Only webhook verification should activate access.

Subscription lifecycle statuses

Use these:

trial

active

past_due

grace_period

expired

cancelled

pending_payment

Grace period rule

Recommended:

7 days after failed renewal or expiry

public content remains live during grace

new uploads and publishing blocked

after grace ends, public properties auto-unpublish

That feels premium and fair.

8. Storage and media architecture

This is where cost control and speed matter.

Storage provider

Cloudflare R2

What is stored there

panorama master files

optimized panorama files

property gallery images

thumbnails

logo images

optional QR exports

future hotspot icons/assets

Storage rule

Do not upload through the backend server itself unless needed for processing.

Best flow:

backend generates signed upload target

frontend uploads directly to R2

This reduces:

Railway bandwidth load

latency

backend memory usage

scaling problems

Recommended bucket structure
users/{userId}/properties/{propertyId}/panorama/master.jpg
users/{userId}/properties/{propertyId}/panorama/optimized.jpg
users/{userId}/properties/{propertyId}/gallery/{imageId}.jpg
users/{userId}/properties/{propertyId}/thumb/cover.jpg
users/{userId}/branding/logo.png
users/{userId}/qr/{propertyId}.png
Storage policy

Enforce total active storage, not monthly storage.
That is much cleaner for users.

File size limits

Recommended hard limits:

Starter: 20 MB per upload

Plus: 25 MB

Pro: 40 MB

Elite: 60 MB

But your real target should be optimized delivery sizes much lower than those.

28. Final end-to-end request flow

Here is the full live system sequence.

A. Sign up

User signs up on app
→ Supabase authenticates
→ profile created
→ default plan state assigned

B. Upgrade

User selects plan
→ frontend calls backend
→ backend initializes Paystack
→ user pays
→ Paystack webhook verifies
→ subscription updated

C. Create property

User creates property metadata
→ property row inserted
→ draft state

D. Upload media

Frontend requests signed upload URL
→ backend checks user + plan + quota
→ signed target returned
→ frontend uploads to R2
→ frontend confirms upload
→ media row stored
→ usage counters updated

E. Publish property

User clicks publish
→ backend checks:

active subscription or grace rules

active property limits

required cover or media

valid slug

→ if valid, property published

F. Public visitor opens page

Visitor opens share link or QR
→ property page loads
→ metadata fetched
→ gallery and/or Pannellum render
→ analytics event stored

G. Lead capture

Visitor submits lead
→ backend validates
→ lead stored
→ owner sees it in dashboard

H. Embed visit

External site iframe loads embed page
→ embed analytics tracked
→ media renders
→ source logged as embed
